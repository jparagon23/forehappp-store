CONVENCIONES
────────────
  ✅  Caso de éxito esperado
  ❌  Caso de error esperado

  PRE: precondiciones requeridas
  ACT: acción a ejecutar
  ESP: resultado esperado
  VAL: validación adicional en BD o respuesta

DIFERENCIAS CLAVE VS CART MODULE:
  - La wishlist NO tiene verificación de rol: cualquier usuario con StoreProfile
    puede usarla (CUSTOMER, SELLER, STORE_ADMIN — sin restricción)
  - Se guarda a nivel de PRODUCTO (no variante)
  - Un usuario tiene exactamente UNA wishlist (OneToOne con StoreProfile)
  - No hay expiración
  - No hay límite de cantidad (es presencia/ausencia de producto)
  - removeItem retorna 200 con WishlistResponse (no 204)
  - clearWishlist retorna 204 (no WishlistResponse)
  - minPrice = precio mínimo entre variantes del producto (0.00 si no tiene variantes)
  - No hay filtrado por ProductStatus — productos INACTIVE aparecen en la wishlist

========================================================
  SECCIÓN 10 — BUGS IDENTIFICADOS EN LA REVISIÓN (TC-251+)
========================================================

BUG-A [ALTO] — N+1 queries al cargar variantes en toItemResponse()
───────────────────────────────────────────────────────────────────
  CÓDIGO: IWishlistRepository hace JOIN FETCH de items y products, pero NO
          de product.variants. En toItemResponse():
            product.getVariants().stream().map(v -> v.getPrice()).min(...)
          Cada llamada dispara una query LAZY de variantes.

  TC-251 ⚠️  GET /wishlist con N ítems genera N+1 queries
    PRE: wishlist con 50 productos
    ESP: 51 queries a BD (1 principal + 50 lazy loads de variants)
    MISMO problema en POST /items y DELETE /items/{id} (ambos llaman toResponse())
    FIX sugerido: agregar LEFT JOIN FETCH i.product p LEFT JOIN FETCH p.variants
                  en la query de IWishlistRepository

BUG-B [MEDIO] — Sin filtro de ProductStatus en la respuesta
───────────────────────────────────────────────────────────────────
  CÓDIGO: toItemResponse() no verifica product.getStatus()

  TC-252 ⚠️  Productos INACTIVE, DRAFT, OUT_OF_STOCK aparecen en la wishlist
    El cart filtra productos no ACTIVE; la wishlist los incluye todos.
    Inconsistencia entre módulos que puede confundir al usuario.
    FIX sugerido: filtrar en toResponse() o marcar en el response con un campo
                  "available": false para que el front muestre el estado.

BUG-C [MEDIO] — Mensaje de error incorrecto en race condition de wishlist
───────────────────────────────────────────────────────────────────
  CÓDIGO: GlobalExceptionHandler maneja DataIntegrityViolationException con:
          "El ítem ya existe en el carrito. Recarga e intenta de nuevo."
          Este mensaje dice "carrito" aunque ocurra en la wishlist.

  TC-253 ⚠️  Concurrent add → 409 con mensaje de "carrito"
    ESP: 409 — body dice "carrito" en el contexto de wishlist
    FIX sugerido: mejorar el handler para detectar el contexto, o capturar
                  DataIntegrityViolationException directamente en el servicio
                  con el mensaje correcto.

BUG-D [BAJO] — clearWishlist llama save() aunque la wishlist esté vacía
───────────────────────────────────────────────────────────────────
  CÓDIGO:
    wishlistDao.findByOwnerId(owner.getId()).ifPresent(wishlist -> {
        wishlist.getItems().clear();  // no-op si ya está vacío
        wishlistDao.save(wishlist);   // escritura innecesaria
    });

  TC-254 ⚠️  DELETE /api/v1/wishlist en wishlist vacía hace escritura en BD
    PRE: wishlist con 0 ítems
    ACT: DELETE /api/v1/wishlist
    ESP: 204 — pero internamente persiste sin cambios
    FIX sugerido: if (!wishlist.getItems().isEmpty()) antes de clear+save

BUG-E [BAJO] — Sin límite de ítems en la wishlist
───────────────────────────────────────────────────────────────────
  TC-255 ⚠️  Usuario puede agregar ilimitados productos a la wishlist
    No hay MAX_ITEMS check en addItem().
    Con 10000 ítems, el response puede ser extremadamente grande.
    FIX sugerido: agregar validación de tamaño máximo (ej: 500 ítems)
                  y retornar 400 si se supera.

BUG-F [INFO] — Orden no determinístico de ítems en la respuesta
───────────────────────────────────────────────────────────────────
  TC-256 ⚠️  El orden de items[] puede variar entre requests
    IWishlistRepository no tiene ORDER BY → orden dependiente de la BD.
    FIX sugerido: agregar ORDER BY i.added_at DESC (más reciente primero)
                  o ORDER BY i.added_at ASC para mantener orden de inserción.

BUG-G [INFO] — variantCount puede quedar desactualizado en caché del front
───────────────────────────────────────────────────────────────────
  TC-257 ⚠️  Si el front cachea la wishlist, variantCount puede ser incorrecto
    minPrice y variantCount se recalculan en cada request, reflejando el estado
    actual del producto. Si el front no refresca, puede mostrar datos obsoletos.
    Recomendación: no cachear el response de wishlist más de 60 segundos.
